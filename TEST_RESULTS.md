测试日期：2026-07-15
环境：Windows PowerShell 7、JDK 17、Android API 36.1 x86_64 模拟器、1080×2400、Debug APK。

- A4：用 1,197,017,687 字节（1.11 GiB）、60 秒、H.264 + AAC 的合成视频执行 LUT 导出成功；应用进程存活且日志无 OOM、Transformer、Muxer 或未处理异常。
- A5（可自动化部分）：下载页空态在系统字体 100% 与 130% 下不越界，底部五项导航完整。
- A6：修复后重复图片/视频 LUT 流程，期间新增崩溃为 0，应用错误扫描为空。
- USB 冷启动静态与生命周期检查：编译后 APK 保留 `USB_DEVICE_ATTACHED`、设备过滤元数据、导出的 `singleTop` MainActivity；强制停止进程后携带 USB action 冷启动 MainActivity 成功，应用恢复到前台且无新增崩溃。真实设备匹配、系统选择框与 MTP 权限仍归 B1–B3。
- 临时 PTP/IP 测试端端到端检查：应用读取到 2 个虚拟相机文件；首次点击提示“已创建下载任务”，活动任务期间再次点击提示“该文件已有下载任务”，列表始终只有一个活动任务。
- 32 MiB 限速视频下载时，下载页同时显示缩略图、圆环百分比、实时速度、已下载/总大小、剩余大小和预计时间，任务卡及底部导航均未越界。
- 活动下载取消后不新增下载记录，任务消失且虚拟 Wi‑Fi 相机会话仍为“已连接”，“进入相册”继续可用。下载中强制结束进程后，重新打开下载页显示中断提示，工作标记自动清除，MediaStore `IS_PENDING` 残留为 0。

## 导出文件核验

- 8 秒 LUT + 90° 视频：MediaStore 时长 8021 ms；ffprobe 为 H.264 8.000 秒、AAC 8.000 秒，带 `rotation=-90` 显示矩阵；解码尺寸为 720×1280。
- 1.11 GiB 输入 LUT 视频：输出 H.264 60.000 秒、AAC 60.011 秒；MediaStore 时长 60032 ms；抽帧颜色与测试 LUT 一致。
- 大文件导出结束后的连续内存采样：Java Heap 回落并稳定在约 14 MB，Native Heap 约 19 MB；导出在首个 5 秒采样前已完成，因此仍需用 Nikon Zf 的真实 4K 大文件观察完整导出期间的峰值。

## 测试中发现并修复

1. 图片和视频 GLSL 原始字符串在 `#version 300 es` 前包含换行，严格驱动会在 GLThread 崩溃。编译入口现统一移除前导空白。
2. `sampler3D` 未声明精度，API 36 严格编译器拒绝编译。图片和视频 Shader 现使用 `highp sampler3D`。
3. GPU 初始化失败原本直接杀进程。现在记录明确错误并安全退出渲染初始化。
4. 分片 MP4 虽可由 ffprobe 播放，但 Android MediaStore/MediaPlayer 将时长识别为 0。导出改为流式 `FrameworkMuxer`，生成标准 MP4 索引且不缓存整段视频字节。
5. LUT 图片下载记录误用了原图大小，现由 MediaStore 回读实际导出大小；视频临时文件写入相册改用 1 MiB 缓冲，减少大文件最终保存阶段的系统调用。
